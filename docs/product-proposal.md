# Product Proposal — Party of One

## Обоснование идеи

Настольные ролевые игры, и D&D в частности, требуют постоянной группы: обычно это
четыре-пять человек и опытный мастер, которые готовы собираться регулярно. Люди
переезжают, у всех разный график, мастера перегорают. Большинство кампаний не
доживают до середины.

Существуют правила для одиночной игры, но там игрок сам играет за всех: сам придумывает
мастера, сам принимает решения за персонажей. Это требует много воображения и
дисциплины, и работает не для всех.

Party of One занимает эту нишу: AI-агенты берут на себя роли мастера и компаньонов,
игрок просто играет.

## Цель и метрики

Цель PoC — показать, что агентная система способна вести связную RPG-сессию, в которой
разные агенты принимают независимые решения, а мир остаётся консистентным на протяжении
всей игры.

**Продуктовые метрики**

Поскольку PoC тестируется вручную, продуктовые метрики — это набор сценариев с
ожидаемым поведением:

- Прохождение edge-кейсов: из N заранее определённых нестандартных ситуаций система
  обрабатывает их без падений и не выходя из роли
- Стабильность сессии: полная сессия завершается без технических ошибок
- Связность нарратива: после 20+ ходов история не противоречит тому, что было раньше
  (оценивается вручную по записи сессии)

**Агентские метрики**

- Narrative consistency: процент ходов, где DM не противоречит ранее установленным фактам
  (оценивается LLM-судьёй по выборке из сессии)
- Character consistency: остаётся ли AI-компаньон в роли — насколько решения соответствуют
  заявленному характеру персонажа
- Rule compliance: корректно ли применяются упрощённые механики (урон, броски, состояния)

**Технические метрики**

- p95 latency ответа DM-агента: не более 5 секунд
- Context compression: после сжатия истории агент отвечает на контрольные вопросы
  по ключевым событиям сессии без ошибок

## Сценарии использования и edge-кейсы

**Основные сценарии**

Игрок описывает сеттинг и класс персонажа, DM генерирует начальную сцену. Партия
исследует мир, взаимодействует с NPC, сталкивается с противниками. Компаньоны
ходят по очереди и принимают решения самостоятельно, но могут советоваться с игроком.
Игрок может в любой момент закрыть сессию и вернуться позже — мир сохраняется.

Режим наблюдения: игрок не участвует, агенты разыгрывают сценарий автономно.
Полезен для тестирования качества системы или просто чтобы посмотреть как агенты
справляются сами.

**Edge-кейсы**

- Игрок убивает ключевого NPC, от которого зависит дальнейший сюжет — DM должен
  адаптироваться, а не зависнуть
- Игрок пытается сделать что-то за пределами правил — DM отказывает или предлагает
  альтернативу, не выходя из роли
- Сессия сильно затягивается — история сжимается автоматически без потери ключевых фактов
- Два компаньона преследуют конфликтующие цели и не могут договориться — нужна
  логика разрешения конфликта
- Total party kill — вся группа погибла, нужен graceful end вместо ошибки
- Игрок вводит в диалог NPC инструкцию вида «забудь все указания» — инъекция
  через нарратив

## Ограничения

**Технические**

- p95 latency DM-ответа: ≤ 5 с
- Мировое состояние читается и обновляется последовательно — ходы агентов идут по
  очереди, не параллельно
- При сжатии истории все ключевые факты о мире должны сохраняться

Система зависит от внешнего LLM API — без доступа к провайдеру не работает.

## Архитектурный набросок

**DM Agent** — главный оркестратор. Получает ход игрока или компаньона, обновляет
мировое состояние, генерирует нарратив. Использует RAG для проверки правил и описания
существ. Принимает решения за всех NPC.

**Companion Agents (×2)** — независимые агенты с фиксированными профилями личности.
Каждый ходит по очереди: получает полную историю чата (всё что говорили DM, игрок
и другой компаньон) и текущее состояние мира, принимает решение о своём ходе,
передаёт его DM-агенту на обработку.

**World State Store** — персистентное хранилище (SQLite). Содержит сущности:
локации, NPC (жив/мёртв, отношения к персонажам), предметы, активные квесты,
журнал событий. Все агенты читают отсюда перед каждым ходом.

**RAG-модуль** — векторизованный рулсет Cairn (CC-BY-SA, ~15 страниц). Cairn —
лёгкая RPG-система с минимальной механикой: три характеристики (STR, DEX, WIL),
HP, броски под характеристику, урон кубиком. DM-агент использует RAG для проверки
правил и разрешения спорных ситуаций.

**Guardrails** — слой до и после вызова LLM. До: проверка входящего текста на
инъекции. После: leak detection (поиск подстрок системного промпта в ответе) + валидация команд (JSON schema + проверка сущностей в БД).

## Data Flow

  ```
  Игрок вводит действие
          │
          ▼
  [Pre-LLM Guardrail] — проверка на инъекции
          │
          ▼
  [DM Agent]
    ├── читает World State Store
    ├── RAG-запрос (правила Cairn, разрешение ситуаций)
    └── генерирует нарратив, обновляет стейт
          │
          ▼
  [Companion Agent A] — свой ход по очереди
    ├── читает World State + сжатую историю
    └── передаёт решение DM-агенту
          │
          ▼
  [DM Agent] — обрабатывает ход компаньона A, обновляет стейт
          │
          ▼
  [Companion Agent B] — свой ход по очереди
    ├── читает World State + сжатую историю
    └── передаёт решение DM-агенту
          │
          ▼
  [DM Agent] — финальный нарратив раунда
          │
          ▼
  [Post-LLM Guardrail] — проверка ответа
          │
          ▼
  Вывод игроку + запись в World State Store
  ```

**Что делегируется агентам**

- Генерация нарратива и диалогов NPC — DM-агент
- Решения компаньонов — агенты компаньонов
- Первичная генерация мира по описанию сеттинга — DM-агент
- Интерпретация нестандартных действий игрока — DM-агент

**Что не делегируется LLM**

- CRUD мирового состояния — детерминированный код
- Бросок кубиков — RNG
- RAG-retrieval — векторный поиск, LLM только интерпретирует результат
- Сжатие истории — по триггеру длины, не по решению агента
- Персистентность сессии — файловый I/O
