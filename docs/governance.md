## Логи и работа с данными

Система не запрашивает и не хранит персональные данные пользователя. Нарративный
контент (диалоги, описания сцен, решения персонажей) хранится локально в SQLite —
никуда не передаётся.

В логи пишутся все запросы к LLM API с промптами и ответами, изменения world state
и ошибки выполнения. Нужны для отладки и для ручной проверки качества сессий.
Хранятся локально, ротации нет.

## Защита от инъекций и контроль действий

**Pre-LLM фильтр** проверяет входящий текст игрока на типовые паттерны инъекции:
прямые команды модели, попытки переопределить системную роль, обращения к «настоящей»
личности агента. При срабатывании запрос отклоняется с игровым ответом DM («твой
персонаж не может сделать это»), а не техническим сообщением об ошибке.

**Post-LLM фильтр** проверяет ответ агента перед выводом. Если в ответе обнаружен
фрагмент системного промпта или невалидная команда — ответ отклоняется
и DM запрашивается повторно с указанием на ошибку.

**Ролевая изоляция**: каждый агент получает свой системный промпт с чёткой ролью.
Пользовательский контент явно отделён от системных инструкций и помечен как
«слова игрока» — модель видит чёткую границу между системными инструкциями и
пользовательским вводом.

**Необратимые игровые действия** (смерть персонажа, конец кампании) фиксируются
в world state только после того, как DM-агент явно подтвердил действие в нарративе.
Прямая запись в стейт минуя DM-агента не предусмотрена.
