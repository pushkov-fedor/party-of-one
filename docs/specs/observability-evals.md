# Spec: Observability & Evals

## Обзор

Логирование и оценка качества. Трек агентский, поэтому в центре — контроль LLM-ответов: не врёт ли нарратив, держат ли компаньоны характер, правильно ли применяются правила Cairn. Технические метрики собираем для отладки, но не они главные.

---

## Логирование

### Что пишем

На каждый LLM-вызов — запись с контекстом:

```json
{
  "ts": "2026-03-31T14:23:01.234Z",
  "event": "llm_call",
  "agent": "dm",
  "turn": 42,
  "round": 14,
  "model": "anthropic/claude-sonnet-4-20250514",
  "prompt_tokens": 4200,
  "completion_tokens": 680,
  "latency_ms": 3200,
  "commands": [
    {"name": "damage_character", "args": {"id": "goblin_1", "amount": 4}, "valid": true}
  ],
  "guardrail_post": "pass",
  "retries": 0,
  "error": null
}
```

На каждый LLM-вызов также пишем `world_state_snapshot` — текстовый снимок мира на момент хода. Нужен для eval pipeline (narrative consistency проверяет ответ DM против состояния мира). Логируется всегда, не только при DEBUG.

Кроме LLM-вызовов логируем:
- Срабатывание guardrails (pre/post, причина блокировки)
- Изменения World State (какая таблица, что было, что стало)
- Компрессию (сколько токенов сжали, сколько фактов дописали из World State)
- Ошибки (тип, сообщение, контекст)
- События сессии (старт, сохранение, восстановление, завершение)

### Формат и хранение

JSON Lines (`.jsonl`), файл `./logs/session.jsonl`. Примерный объём — ~1 MB на сессию из 30 раундов. Ротации нет (PoC).

При уровне `DEBUG` пишутся полные промпты и ответы — нужно для eval pipeline. При `INFO` — только метаданные.

## Метрики

### Технические (автоматически)

| Метрика | Зачем |
|---------|-------|
| **Error rate** | Доля упавших LLM-вызовов. Если > 5% — проблема с провайдером или промптами |
| **Retry rate** | Доля вызовов, потребовавших retry. Если > 10% — промпты или guardrails слишком строгие |
| **Guardrail block rate** | Как часто guardrails блокируют ответ. Для отладки порогов |
| **Стоимость сессии** | Сумма токенов × цена. Ориентир — ≤ $3 за 30 раундов |

Считаются по логам после сессии. В runtime не нужны.

### Агентские (LLM-as-judge, после сессии)

Это основные метрики проекта. Оцениваются по логам через watch mode.

| Метрика | Что проверяем | Целевое значение |
|---------|--------------|-----------------|
| **Консистентность нарратива** | DM не противоречит World State | ≥ 90% ходов без противоречий |
| **Консистентность персонажей** | Компаньоны не выпадают из роли | ≥ 85% ходов в характере |
| **Соблюдение правил** | Механики Cairn применены корректно | ≥ 95% корректных бросков |
| **Качество retrieval** | Нужные правила попали в top-3 | ≥ 90% hit rate на тестовых запросах |
| **Качество компрессии** | Ключевые факты не потеряны при сжатии | ≥ 90% фактов сохранены |

## Eval pipeline

### Консистентность нарратива

Берём выборку ходов DM из лога, каждый прогоняем через LLM-judge:

```
Ты — судья, оценивающий консистентность нарратива в RPG-сессии.

## Установленные факты (World State на момент хода)
{world_state_snapshot}

## Предыдущие события
{compressed_history + recent_turns}

## Ответ DM
{dm_response}

## Вопрос
Содержит ли ответ DM противоречия с установленными фактами?
Ответь JSON: {"consistent": true/false, "issues": ["описание проблемы"]}
```

Итог: доля ходов с `consistent=true`.

### Консистентность персонажей

Все ходы компаньонов из лога:

```
## Профиль персонажа
{personality_profile}

## Ситуация
{context}

## Действие персонажа
{companion_action}

## Вопрос
Соответствует ли действие характеру персонажа?
Ответь JSON: {"in_character": true/false, "explanation": "..."}
```

Итог: доля `in_character=true`.

### Соблюдение правил

Все ходы с командой `roll_dice`:

```
## Правила Cairn (релевантные)
{rag_chunks_for_situation}

## Ситуация
{context}

## Действие DM
{dm_response + commands}

## Вопрос
Корректно ли применены правила? Правильный ли кубик? Правильно ли обработан результат?
Ответь JSON: {"correct": true/false, "issues": ["..."]}
```

Итог: доля `correct=true`.

### Качество retrieval

Детерминированная проверка, без LLM. Заранее готовим набор из 10–15 тестовых запросов с ожидаемыми секциями правил:

| Запрос | Ожидаемая секция |
|--------|-----------------|
| «атака мечом» | Combat / Damage |
| «проверка WIL» | Saves |
| «критический урон» | Critical Damage |
| «лечение раненого» | Healing / Rest |
| ... | ... |

Прогоняем каждый запрос через retriever, проверяем что нужный чанк попал в top-3 результатов. Итог: hit rate — доля запросов, где правильный чанк нашёлся.

Если hit rate низкий — проблема либо в chunking (чанки слишком крупные/мелкие), либо в формировании запроса (ключевые слова не совпадают с текстом правил).

### Качество компрессии

Без LLM, детерминированная проверка:
1. Берём историю до компрессии
2. Извлекаем ключевые факты из дельт World State (смерти, квесты, перемещения)
3. Проверяем, упомянут ли каждый факт в сжатой истории
4. Итог: доля найденных фактов

## Тестовые сценарии (ручные)

| Сценарий | Ожидание | Проверка |
|----------|----------|----------|
| Игрок убивает ключевого NPC | DM адаптирует сюжет | Нет crash, нарратив продолжается |
| Действие вне правил | DM отказывает в рамках роли | Нет выхода из роли |
| Длинная сессия (20+ ходов) | Компрессия работает | Нет переполнения контекста |
| Конфликт компаньонов | DM разрешает в нарративе | Нет бесконечного цикла |
| Вся партия погибла | Финальная сцена | Корректное завершение |
| Prompt injection | Pre-LLM блокирует | Нет выхода из роли |

## Автоматический прогон (watch mode)

Запуск полной сессии без человека. По завершении — LLM-judge eval-ы по логу.

```bash
python -m party_of_one.eval \
  --mode watch \
  --rounds 30 \
  --scenario "dark_fantasy_dungeon" \
  --output eval_results.json
```
